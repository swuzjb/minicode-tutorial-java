Chapter 07

Context 管理

用了一阵子，API 返回 400: context_length_exceeded。一个大 pom.xml 就是几千 token。

你让 MiniCode 帮你重构一个 Maven 多模块项目。它先读了几个 pom.xml，然后看了十来个 Java 文件，执行了几次 mvn 命令。一切顺利，直到第 20 轮对话——API 突然返回 400 错误。

原因很简单：每次 API 调用都要发送完整的对话历史。工具返回的文件内容、命令输出都在里面。一个 pom.xml 几千 token，一段 mvn 输出上万 token。累积起来很快就撞到 200K 的上限。

关键洞察：不需要 tokenizer

你可能想"用 tokenizer 计算每条消息的 token 数"。不用。Anthropic API 每次响应都告诉你这次请求用了多少 input token：

// response.usage() 直接告诉你
long inputTokens = response.usage().inputTokens();
// 这就是当前 context 的实际大小，精确到 token

这是最精确的度量。不需要本地 tokenizer，不需要猜测，API 直接告诉你。

Token 追踪

在 agent loop 中追踪 context 使用率：

private static long lastInputTokens = 0;
private static final long MAX_CONTEXT = 200_000;

// 每次 API 调用后更新
lastInputTokens = response.usage().inputTokens();
var utilization = (double) lastInputTokens / MAX_CONTEXT;
showContextBar(utilization);

策略一：压缩工具返回

最大的 token 消耗者通常是工具返回值。一个 cat pom.xml 返回的内容可能有 3000 token。但在对话后期，Claude 并不需要完整内容——它已经"看过"了。我们可以截断旧的工具返回：

private static List<MessageParam> compressToolResults(List<MessageParam> messages) {
    return messages.stream().map(msg -> {
        // 只处理 tool_result 类型的消息
        if (!isToolResult(msg)) return msg;

        var content = getToolResultText(msg);
        var lines = content.split("\n");
        if (lines.length <= 80) return msg;

        // 保留前 60 行 + 后 20 行，中间用省略标记
        var head = String.join("\n", Arrays.copyOfRange(lines, 0, 60));
        var tail = String.join("\n", Arrays.copyOfRange(lines,
            lines.length - 20, lines.length));
        int omitted = lines.length - 80;
        var compressed = head + "\n[... " + omitted + " lines omitted ...]\n" + tail;

        return replaceToolResultText(msg, compressed);
    }).toList();
}

策略二：丢弃旧消息

如果压缩还不够，最后的手段是丢弃对话早期的消息。保留最近 N 轮对话，前面的用一句话概括：

private static List<MessageParam> dropOldMessages(
        List<MessageParam> messages, int keepRecent) {
    if (messages.size() <= keepRecent) return messages;

    var notice = MessageParam.ofUser(
        "[Earlier messages removed to free context]");
    var recent = messages.subList(
        messages.size() - keepRecent, messages.size());
    var result = new ArrayList<MessageParam>();
    result.add(notice);
    result.addAll(recent);
    return result;
}

集成到 Agent Loop

在每次 API 调用前检查 context 使用率，渐进式触发压缩策略：

// 在 agent loop 的每次迭代中
if (lastInputTokens > MAX_CONTEXT * 0.8) {
    System.out.println("  [context] Compressing tool results...");
    messages = compressToolResults(messages);
}
if (lastInputTokens > MAX_CONTEXT * 0.9) {
    System.out.println("  [context] Dropping old messages...");
    messages = dropOldMessages(messages, 10);
}

两级策略：80% 时压缩工具输出（温和），90% 时丢弃旧消息（激进）。这样大多数情况下用户感知不到 context 管理的存在。

CLI 进度条

让用户直观地看到 context 使用情况。用 ANSI 转义码画一个彩色进度条：

private static void showContextBar(double utilization) {
    int width = 30;
    int filled = (int) (utilization * width);
    String color = utilization > 0.9 ? "\033[31m"   // red
                 : utilization > 0.7 ? "\033[33m"   // yellow
                 : "\033[32m";                        // green

    String bar = color
        + "\u2588".repeat(filled)
        + "\033[90m"
        + "\u2591".repeat(width - filled)
        + "\033[0m";
    System.out.printf("  [%s] %.0f%% context used%n", bar, utilization * 100);
}

进阶方向

更高级的 context 管理策略包括：用 Claude 自己来总结旧对话（消耗一次 API 调用但压缩率更高）、按工具返回的重要性评分、或者给不同类型的内容设置不同的 TTL。这些留给有兴趣的读者作为练习。
