Chapter 05

安全防护

bash 工具可以执行 rm -rf /。Claude 幻觉一次你就完了。

试试问 MiniCode "删除当前目录所有 .java 文件"。它会毫不犹豫地调用 bash 工具执行 rm *.java。你甚至来不及眨眼，源代码就没了。

这不行。

AI 模型会犯错。它可能误解你的意图，可能产生幻觉，可能在推理过程中走偏。当它手里握着一个不受约束的 bash 工具时，一次犯错的代价是不可逆的文件丢失。我们需要在工具执行层加入防护。

最简方案——确认机制

最直接的思路：每次执行写操作或命令前，问用户一声。

private static boolean confirm(String message) {
    System.out.print("\n⚠️  " + message + " [y/N] ");
    var answer = new Scanner(System.in).nextLine().trim().toLowerCase();
    return answer.equals("y");
}

够简洁。Java 里一个 Scanner 就搞定。无需回调、无需 CompletableFuture——阻塞式 IO 在 CLI 里反而是优势。

修改执行流程

把确认逻辑嵌入 Registry 的 execute 方法。读操作直接放行，写操作和命令执行需要用户点头：

public String execute(String name, JsonNode input) {
    var tool = tools.get(name);
    if (tool == null) return "Error: unknown tool \"" + name + "\"";

    // 读操作直接执行
    if (name.equals("read_file") || name.equals("grep")) {
        return tool.execute(input);
    }

    // 写操作/命令需要确认
    if (!confirm("Allow " + name + "(" + input + ") ?")) {
        return "User denied this operation.";
    }

    return tool.execute(input);
}

好，现在至少不会悄无声息地执行危险命令了。但用了一天你就会发现新问题。

这够了吗？

每次都确认太烦了。mvn test 需要确认吗？git status 需要确认吗？ls 呢？如果每个命令都弹确认框，你会疯掉，然后肌肉记忆地按 y——这时候确认机制就形同虚设了。

我们需要规则：哪些自动允许，哪些需要确认，哪些直接拒绝。

权限规则引擎

定义三种权限级别，然后用规则列表匹配。Java 21 的 record 和 sealed 类型非常适合建模这种结构：

enum Permission { ALLOW, ASK, DENY }

record PermissionRule(
    String tool,
    Permission permission,
    Predicate<JsonNode> condition
) {
    // 无条件规则的便捷构造
    PermissionRule(String tool, Permission permission) {
        this(tool, permission, null);
    }
}

然后定义规则列表——顺序很重要，第一个匹配的规则生效：

private static final List<PermissionRule> RULES = List.of(
    // 读操作：自动允许
    new PermissionRule("read_file", Permission.ALLOW),
    new PermissionRule("grep", Permission.ALLOW),

    // 危险命令：直接拒绝（注意：deny 规则要在 allow 前面）
    new PermissionRule("bash", Permission.DENY, input -> {
        var cmd = input.get("command").asText();
        return cmd.matches(".*rm\\s+-rf.*|.*sudo.*|.*mkfs.*");
    }),

    // 安全的命令：自动允许
    new PermissionRule("bash", Permission.ALLOW, input -> {
        var cmd = input.get("command").asText();
        return cmd.matches("ls.*|cat.*|pwd|git status|git diff|mvn test.*");
    }),

    // 其他：询问用户
    new PermissionRule("*", Permission.ASK)
);

权限检查函数

遍历规则列表，找到第一个匹配的规则：

private static Permission checkPermission(String toolName, JsonNode input) {
    for (var rule : RULES) {
        // 工具名不匹配，跳过
        if (!rule.tool().equals("*") && !rule.tool().equals(toolName)) continue;
        // 有条件但条件不满足，跳过
        if (rule.condition() != null && !rule.condition().test(input)) continue;
        // 匹配！返回权限
        return rule.permission();
    }
    return Permission.ASK;
}

逻辑清晰：逐条匹配，第一个命中就返回。通配符 * 兜底。Java 的 record 让这段代码比传统 POJO 简洁了一倍不止。

"Always allow" 记忆

频繁确认还是烦。比如你让 MiniCode 帮你重构，它需要写十几个文件。每次都按 y 太累了。我们加一个 a（always）选项——一旦用户选择 always，同类操作就不再询问：

private static final Set<String> alwaysAllowed = new HashSet<>();

private static boolean authorize(String name, JsonNode input) {
    // 已标记为 "always allow" 的工具直接放行
    if (alwaysAllowed.contains(name)) return true;

    var permission = checkPermission(name, input);
    if (permission == Permission.ALLOW) return true;
    if (permission == Permission.DENY) {
        System.out.println("\n🚫 Denied: " + name + " — matches a blocked pattern");
        return false;
    }

    // ASK：询问用户
    System.out.print("\n⚠️  Allow " + name + "? [y/n/a(always)] ");
    var answer = new Scanner(System.in).nextLine().trim().toLowerCase();
    if (answer.equals("a")) {
        alwaysAllowed.add(name);
        return true;
    }
    return answer.equals("y");
}

安全边界的诚实评估

局限性

正则匹配不能防住所有攻击。命令混淆（r\m -rf /）、变量替换（$CMD）、管道组合（find / -delete）都能绕过简单的模式匹配。生产环境应考虑容器隔离（Docker）或沙箱（如 gVisor）。Java 生态还可以考虑用 SecurityManager 的替代方案或 ProcessBuilder 的受限执行。

但对一个学习项目来说，这已经是合理的防护层了。它能拦截绝大多数"Claude 脑子一热"的情况。

本章成果

MiniCode 现在不会自己删你的文件了。读操作秒过，安全命令自动放行，危险命令被拦截。对于介于中间的操作，用户可以逐次确认或选择 "always allow"。下一章我们让别人也能给它加工具——通过 MCP 协议。
